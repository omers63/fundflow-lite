AccountingService::safeDeleteLoan(Loan $loan)
Inside a single DB transaction it:

Collects all account_transactions whose source is this loan (disbursement, imported bulk repayments) or any of its installments (fund repayments, cash debits, guarantor debits).
Reverses each line with the existing applyLedgerEntryReversal helper (balances stay consistent).
Picks up any remaining lines on the loan’s virtual account (orphan/straggler rows) and reverses those too.
Deletes all installments for the loan.
Deletes the loan Account row (type = loan, loan_id set).
Deletes the loan record.

# Partial Loan Disbursement & Master Fund Guard

## Background

Currently the `disburse` action executes a single atomic disbursement of the full approved amount, builds the full installment schedule immediately, and sets loan status to `active`. The master fund balance is not checked before debiting.

The required change introduces:
1. **Master fund cannot go negative** — hard block on any debit that would push it below zero.
2. **Partial disbursements** — the admin can disburse in portions as master fund capacity allows.
3. **Repayment only after full disbursement** — installments are only generated once the total disbursed equals `amount_approved`.
4. **Contribution exemption during approval-to-full-disbursement** — member stays exempt while `approved` (already true) AND while partially disbursed (currently we move to `active` at disbursement start — this needs fixing).

---

## User Review Required

> [!IMPORTANT]
> A new `loan_disbursements` table will be created to track each partial disbursement event. This is the cleanest approach for auditability.

> [!WARNING]
> The `disbursed_at` column on `loans` will now store the date of **final (full) disbursement**, not the first. Partial disbursements are tracked in `loan_disbursements`.

> [!IMPORTANT]
> **Status flow change:** A loan that has been partially (but not fully) disbursed will remain in `approved` status until the last portion is disbursed. Only then does it transition to `active` and generate the installment schedule.

> [!CAUTION]
> **Existing `active` loans are unaffected** — they were fully disbursed in a single shot. The new columns (`amount_disbursed`) will be filled from migrations with their `amount_approved` value so the `is_fully_disbursed` property works correctly for them.

Here is what was implemented so every ledger row (account_transactions) has a polymorphic source.

1. AccountingService
postEntry now requires an Illuminate\Database\Eloquent\Model as source and stores source_type / source_id via getMorphClass() and getKey().
Member CSV import adjustments use the Member as source (same rows as today, but traceable).
Parent → dependent cash transfer: debit line sources $parent, credit line sources $dependent.
postManualLedgerEntry uses the User who posted (posted_by / auth()->id()), with User::findOrFail(1) as fallback only if that user is missing.
postContribution runs in one transaction: if payment_method === Contribution::PAYMENT_METHOD_CASH_ACCOUNT, it calls debitMemberCashForContribution (cash lines use the Contribution as source), then posts the two fund credits (unchanged, still sourced to the contribution).
Removed debitCashForContribution; cash for cycle contributions is handled inside postContribution.
2. Contribution model
Added PAYMENT_METHOD_CASH_ACCOUNT = 'cash_account' so the cash leg is keyed off a real constant.
3. ContributionCycleService
Stops debiting cash before create; Contribution::create with cash_account lets the observer run postContribution, which debits cash and credits funds with proper sources.
4. QuickPostWorkflowService
Drops the extra debit + duplicate postContribution.
Creates the contribution with payment_method = PAYMENT_METHOD_CASH_ACCOUNT so the observer does cash + fund once (still inside the existing DB::transaction).
5. Migration 2026_04_12_120000_backfill_and_require_account_transaction_source.php
Backfills any rows with source_type null (including soft-deleted ledger rows), using description/member_id/posted_by heuristics, then source_type / source_id are NOT NULL.
6. UI
Ledger filter adds source types Member and User.
Run migrations where this hasn’t been applied yet:

php artisan migrate

Note: Deleting a contribution still reverses all lines whose source is that contribution (cash deduction + fund credits). Quick-post contributions are now stored as cash_account (not bank_transfer) so that behavior stays correct.
