AccountingService::safeDeleteLoan(Loan $loan)
Inside a single DB transaction it:

Collects all account_transactions whose source is this loan (disbursement, imported bulk repayments) or any of its installments (fund repayments, cash debits, guarantor debits).
Reverses each line with the existing applyLedgerEntryReversal helper (balances stay consistent).
Picks up any remaining lines on the loan’s virtual account (orphan/straggler rows) and reverses those too.
Deletes all installments for the loan.
Deletes the loan Account row (type = loan, loan_id set).
Deletes the loan record.

# Partial Loan Disbursement & Master Fund Guard

## Background

Currently the `disburse` action executes a single atomic disbursement of the full approved amount, builds the full installment schedule immediately, and sets loan status to `active`. The master fund balance is not checked before debiting.

The required change introduces:
1. **Master fund cannot go negative** — hard block on any debit that would push it below zero.
2. **Partial disbursements** — the admin can disburse in portions as master fund capacity allows.
3. **Repayment only after full disbursement** — installments are only generated once the total disbursed equals `amount_approved`.
4. **Contribution exemption during approval-to-full-disbursement** — member stays exempt while `approved` (already true) AND while partially disbursed (currently we move to `active` at disbursement start — this needs fixing).

---

## User Review Required

> [!IMPORTANT]
> A new `loan_disbursements` table will be created to track each partial disbursement event. This is the cleanest approach for auditability.

> [!WARNING]
> The `disbursed_at` column on `loans` will now store the date of **final (full) disbursement**, not the first. Partial disbursements are tracked in `loan_disbursements`.

> [!IMPORTANT]
> **Status flow change:** A loan that has been partially (but not fully) disbursed will remain in `approved` status until the last portion is disbursed. Only then does it transition to `active` and generate the installment schedule.

> [!CAUTION]
> **Existing `active` loans are unaffected** — they were fully disbursed in a single shot. The new columns (`amount_disbursed`) will be filled from migrations with their `amount_approved` value so the `is_fully_disbursed` property works correctly for them.

Here’s a practical way to think about caching for this app (Laravel + Filament admin, heavy widgets, aggregates).

1. Production Laravel basics (big wins, low risk)
php artisan config:cache and route:cache in production so every request isn’t reparsing config/routes.
php artisan view:cache if you ship compiled Blade (Filament uses a lot of views).
composer install --optimize-autoloader --no-dev on deploy.
These don’t change your code; they cut bootstrap cost.

2. Cache expensive read-mostly aggregates (fits your widgets)
Places like MemberStatsWidget::getData() and banking overview widgets run many count() / whereHas() / avg() queries on every load. Good pattern:

Wrap the whole getData() result (or each logical block) in Cache::remember($key, $ttl, fn () => ...).
Key by something that changes when data changes, e.g. member_stats:v1 plus a version or max(updated_at) from members (cheap query) so you don’t serve stale numbers forever without invalidation.
Invalidate (or bump version) when members, contributions, loans, or installments change—e.g. in model observers, event listeners, or after Filament actions that mutate those tables.
TTL alone (e.g. 60–300 seconds) is the minimal approach if you don’t want event wiring yet; it’s simple but can show slightly stale numbers.

Store: use Redis or Memcached in production if you can; file/database drivers work but add I/O and don’t scale as well for many keys.

3. Cache::tags() (Redis only)
If you use Redis, tag caches like ['members', 'contributions'] and Cache::tags(...)->flush() when those domains change. Without tags, you either use TTL, manual forget, or a global “stats version” integer in cache/DB that you increment on writes.

4. Per-user vs global
Global dashboards (totals, charts): cache one payload for all admins (key does not include user_id), invalidate on data changes.
Permission-sensitive fragments: if the cached blob differs by role, include a role or permission hash in the key, or keep those bits uncached.
5. Filament / Livewire specifics
Full page admin UI is mostly Livewire; you typically don’t cache HTML. You cache data the widgets compute.
databaseNotificationsPolling('30s') causes recurring requests; caching doesn’t replace that, but ensuring notification queries are indexed and cheap helps. Reducing poll interval or using broadcasting for “new notification” is a different lever than Cache::remember.
After mutations, you already refresh widgets in places (dispatchMemberListHeaderWidgetsRefresh); you can Cache::forget(...) (or bump version) in the same place so the next render recomputes once.
6. What not to cache blindly
Per-row Filament table data: caching is awkward because of filters, search, sort, and pagination; you already improved that with eager aggregates. Prefer indexes + query shape there.
User-specific mutable state without a clear invalidation story.
7. Optional advanced pieces
HTTP cache (Cache-Control) for public assets and any anonymous routes; admin sessions rarely benefit.
Laravel Octane reduces bootstrap cost per request; it’s ops-heavy and not the same as Cache::remember, but it pairs well with in-memory caches.
Minimal first step for your codebase: add Cache::remember around MemberStatsWidget::getData() (short TTL, e.g. 120s), then add explicit forget / version bump on member/contribution/loan writes when you’re ready. If you want this implemented in-repo next, say which widgets and TTL/invalidation style you prefer (TTL-only vs Redis tags vs max(updated_at) keys).

You can get something that feels like a Windows 11 app, but there is no single button that turns this Laravel + Filament stack into a normal .exe the way a WinForms or WPF project works. The app is a server-rendered web app (PHP, DB, queues, etc.), so a “standalone” desktop build is really about how you host and wrap that stack.

Realistic approaches
Installed web stack + app shell (most common)
Ship PHP + your code + SQLite/MySQL (or use Laragon/XAMPP-style layout) and a small launcher that:

starts php artisan serve (or nginx/php-fpm),
opens Edge in app mode or a WebView2 window pointed at http://127.0.0.1:....
The “Windows app” is the launcher + embedded browser; the product is still your Laravel app.
Electron / Tauri + your URL
A thin desktop window that loads only your site (e.g. https://your-domain/admin).
Simple to ship, but it’s not offline and not “standalone” unless the backend runs locally (see option 1).

PWA (installable site)
In Edge: install site as app → icon on taskbar, separate window.
Good UX, minimal work, still needs network (or local server) for Laravel.

NativePHP for Laravel
Projects that package Laravel into a desktop binary with an embedded PHP runtime exist (e.g. NativePHP). This is the closest to “one executable” for Laravel, but you still need to handle database, updates, signing, antivirus, and Filament’s expectations (assets, storage, permissions). It’s a dedicated project, not a quick export.

Full rewrite (UWP / WinUI / .NET MAUI)
Rebuild UI and business rules in another stack. Not derived automatically from this codebase.

If “standalone” means runs on a PC without installing Laragon manually: plan for local server + WebView2/Electron launcher or evaluate NativePHP with your constraints (DB, backups, updates).
If “standalone” means icon on Windows that opens FundFlow: PWA or Edge app + your hosted URL is the fastest.

Design (decisions baked in)
What counts as a “miss” (per closed cycle month)
After the cycle deadline for month M has passed:

Contribution: required if the member is not contribution‑exempt that month (no approved/active loan with approved_at on/before month end), monthly_contribution_amount > 0, and there is no contribution row for (month, year).
Repayment: any active loan has an installment with due_date in that month that is still pending or overdue.
A month counts as one miss if either applies (not double‑counted).
Consecutive threshold
Trailing consecutive closed months with a miss, counting backward from the latest closed period (not “max streak ever”).

Total threshold
Number of miss‑months in a rolling lookback (default 60 months), not lifetime—so totals can improve as old months drop out of the window.

That command:

Marks pending installments whose due_date is in the past as overdue

It’s also scheduled daily at 08:00 (see routes/console.php). To run it on a schedule in production you need a cron entry that runs Laravel’s scheduler, e.g.:

Here’s what “backfill” means here and how to do it safely.

What you’re backfilling
The delinquency job treats a miss as: for a closed month, a contribution was required but there is no contributions row for that (member_id, month, year) (and/or loan installments are still unpaid).

Backfill = put the historical contribution history into the app so those months are not counted as empty.

How to backfill contributions
1. CSV import (bulk)
Admin: Finance → Contributions (or your Contributions resource), use Import contributions and upload a CSV.

Columns: member_id or member_number, plus month, year, amount.
Optional: paid_at, reference_number, notes, is_late, payment_method.
Format matches ContributionImportService (see docs/thinking.txt around the Contribution import section, or samples/contributions-import-sample.csv if present).
2. One row per member per month
You cannot have two active rows for the same member + month + year; duplicates are rejected.

3. After data is in
Run:

php artisan fund:check-delinquency
so suspension/restore lines up with the updated history.

Accounting warning (important)
Saving a contribution runs ContributionObserver → AccountingService::postContribution() — it posts to the ledger, not only a label. So:

Treat backfill as real financial history, or
Coordinate with whoever maintains the books so totals stay correct, or
If you only need to avoid false delinquency without moving money again, you may need a separate business process (e.g. journal-only adjustments, or temporarily higher delinquency thresholds in System settings → Contribution cycles → Delinquency policy) until history is complete.
Loan repayments
“Misses” on loan installments are not fixed by contribution CSV. Those need the installment paid (normal repayment run, admin posting, or whatever your app uses for repayments). Once installments are paid, those months stop counting as repayment misses.

If people were suspended too aggressively
After backfilling and re-running fund:check-delinquency, members who are now under the thresholds should be auto-restored if they were suspended only for delinquency (delinquency_suspended_at set). If something is still wrong, manually set status in Members as your policy allows.

Summary: Backfill = import or create historical contributions (one per member per month that should count as paid), understand that creates ledger postings, then run php artisan fund:check-delinquency. For loans, clear installments through repayment flows, not the contribution importer.

A new "Refund" header action appears only on member cash accounts (top of the ledger table). It:

Pre-fills with the full current balance (editable down to a partial amount)
Calls AccountingService::refundMemberCash() which posts a debit to both the member cash account and master cash in a single DB transaction
The matching debit will appear on the next imported bank statement for reconciliation
3. Annual Subscription Fees
New member_subscription_fees table (one row per member per year, unique constraint)
New MemberSubscriptionFee model with scopes
SubscriptionFeeService::chargeMember() posts the fee to master cash (credit only, like late fees and membership fees)
SubscriptionFeeService::chargeAnniversaryFees() auto-charges all active members whose join-date anniversary falls today
Daily scheduler entry at 06:00 in routes/console.php
"Charge Annual Subscription" single-record action on each member row in MemberResource
Bulk action on the members list page to charge multiple members at once
Setting::annualSubscriptionFee() helper; configurable in System Settings → Cycle Settings under a new "Annual Subscription Fee" section
4. Fee Revenue Dashboard Widget
A new full-width widget (FeesRevenueWidget) appears on the admin dashboard (sort=3, after the Financial Health widget). It shows three side-by-side cards:

Late Fees (red accent) — all-time total, this-year breakdown with progress bar
Membership Fees (sky accent) — application fees posted to master cash
Annual Subscription Fees (emerald accent) — or an empty-state prompt to configure the fee if none exist yet
